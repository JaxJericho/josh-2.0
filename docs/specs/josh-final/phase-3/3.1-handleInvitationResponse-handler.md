## Implementation

### File: `packages/messaging/src/handlers/handle-invitation-response.ts`

#### Local Parse Rules

Parse runs before any LLM call, inside the router — but document the token set here for the implementation.

Accept tokens: `yes`, `y`, `in`, `i'm in`, `count me in`, `ok`, `sure`, `sounds good`, `1`
Pass tokens: `no`, `n`, `pass`, `skip`, `not this time`, `can't`, `cant`, `nope`, `2`

Any other input: send one clarification prompt. Set a `clarifier_pending` state token on the session to prevent a loop. If the next message is also unrecognized after clarification, treat as `pass`.

#### Function Signature

```ts
export async function handleInvitationResponse(
  context: HandlerContext,
): Promise<HandlerResult>;
```

#### On `accept`

1. Load the pending invitation:
   ```sql
   SELECT * FROM invitations
   WHERE user_id = $userId
     AND state = 'pending'
     AND expires_at > now()
   ORDER BY created_at DESC
   LIMIT 1
   ```
2. **Idempotency check**: If `invitation.response_message_sid` is already set and equals `context.messageSid`, return the previously sent response without re-applying side effects.
3. If no pending invitation exists (expired or not found): respond with `"It looks like that invitation has already expired. JOSH will be in touch with something new soon."` Set session mode to `idle`. Return.
4. Inside a single transaction:
   - Set `invitations.state = 'accepted'`, `responded_at = now()`, `response_message_sid = context.messageSid`
   - Write to `learning_signals`:
     ```ts
     {
       user_id: context.userId,
       signal_type: 'invitation_accepted',
       payload: {
         invitation_id: invitation.id,
         invitation_type: invitation.invitation_type,
         activity_key: invitation.activity_key,
         proposed_time_window: invitation.proposed_time_window,
       },
       correlation_id: context.correlationId,
     }
     ```
   - Set `users.invitation_backoff_count = 0`
   - Set `conversation_sessions.mode = 'idle'`
5. If `invitation.invitation_type = 'solo'`: enqueue outbound SMS with solo confirmation copy (see copy spec). Return.
6. If `invitation.invitation_type = 'linkup'`: call `evaluateLinkupQuorum(invitation.linkup_id)` (Phase 6, Ticket 6.8 — import the function; it may not exist yet when this ticket runs). Enqueue outbound SMS with linkup acceptance copy. Return.

**Dependency note**: `evaluateLinkupQuorum` does not exist yet when Ticket 3.1 runs (Phase 6 implements it). Import it from `packages/core/src/invitation/linkup-quorum.ts` and stub the call behind an existence check or a feature flag. The integration test for the full accept→lock path runs in Phase 8. Do not block the ticket on Phase 6 completion.

#### On `pass`

1. Load the pending invitation (same query as above — no `expires_at` filter; a user can pass an expired invitation gracefully).
2. **Idempotency check**: same as accept.
3. If no invitation found: respond with `"No worries — JOSH will find something else for you."` Set mode to `idle`. Return.
4. Inside a single transaction:
   - Set `invitations.state = 'passed'`, `responded_at = now()`, `response_message_sid = context.messageSid`
   - Write to `learning_signals`:
     ```ts
     {
       user_id: context.userId,
       signal_type: 'invitation_passed',
       payload: {
         invitation_id: invitation.id,
         invitation_type: invitation.invitation_type,
         activity_key: invitation.activity_key,
         proposed_time_window: invitation.proposed_time_window,
         // pass_reason and time_hint are added by Ticket 7.1
       },
       correlation_id: context.correlationId,
     }
     ```
   - Increment `users.invitation_backoff_count` by 1
   - Set `conversation_sessions.mode = 'idle'`
5. Enqueue outbound SMS: `"Got it — no problem. JOSH will keep looking."` Return.

#### SMS Copy

Solo confirmation:

> "You're set for [activity_display_name] [proposed_time_window]. JOSH will check in with you afterward. Reply STOP anytime to unsubscribe."

Linkup acceptance:

> "You're in for [activity_display_name] [proposed_time_window]. JOSH will confirm the group and send details once everyone is locked in."

Use `activity_catalog` to resolve `activity_key` → `activity_display_name`. If the catalog lookup fails, fall back to `activity_key` formatted as title case.

#### Clarification copy

> "Reply YES to accept or PASS to skip."

## Router Update

In `supabase/functions/_shared/router/conversation-router.ts`, add the `awaiting_invitation_response` local-parse shortcut. This must run before the LLM classifier:

```ts
// Local parse for invitation responses — runs before LLM
if (session.mode === "awaiting_invitation_response") {
  const parsed = parseInvitationResponse(messageText); // local token parse
  if (parsed === "accept" || parsed === "pass" || parsed === "clarify") {
    return handleInvitationResponse(context);
  }
}
```

The local parser (`parseInvitationResponse`) is a pure function defined in `handle-invitation-response.ts` and exported for the router to import.

## Hard Rules for This Ticket

- Every state transition — `pending → accepted`, `pending → passed` — executes inside a transaction. No partial writes.
- Idempotency on MessageSid is non-negotiable. Twilio retries webhooks on timeout; the handler must be safe to call twice.
- Do not write a learning signal on the idempotency early-return path.
- Do not call `evaluateLinkupQuorum` inline if the import does not resolve — stub with a `TODO` comment and a logged warning. Phase 8 integration tests cover the full accept→lock path.
- `SUPABASE_SERVICE_ROLE_KEY` is server-side only. This handler runs in a Supabase edge function — confirm the client initialization follows the existing pattern.
- Set mode to `idle` in every terminal branch (accept, pass, expired-not-found, clarification timeout). Never leave `awaiting_invitation_response` as the terminal mode.
