# Ticket 3.0 — Invitation Response Atomicity Foundation

## Why This Ticket Exists

Ticket 3.1 requires invitation accept/pass handling to be atomic and MessageSid-idempotent. Current `main` does not provide a DB-level replay-safe guarantee for `invitations.response_message_sid`, and there is no existing transactional primitive dedicated to invitation response side effects.

This ticket creates the minimal DB foundation required so Ticket 3.1 can be implemented correctly without relying on race-prone TypeScript-only read/check/write logic.

## Scope

Add the smallest DB-side foundation required for invitation-response atomicity and replay safety:

1. Add a DB-level uniqueness guarantee for `public.invitations.response_message_sid`
2. Add a transactional RPC that performs the invitation response state transition and all required side effects in one transaction

This ticket is foundation only. It does not implement the invitation-response handler, router wiring, classifier changes, or user-facing SMS copy beyond what is strictly required by the RPC contract.

## Files In Scope

- `supabase/migrations/<timestamp>_ticket_3_0_invitation_response_atomicity_foundation.sql`
- `supabase/types/database.ts`
- `packages/db/src/types.ts` or any committed DB type surface used by this repo
- Minimal tests if the repo already has a pattern for exercising new RPC contracts
- No unrelated runtime files unless required for compile-safe type exposure

## Required DB Changes

### 1) Replay-Safe Constraint

Add a DB-level uniqueness guarantee for `public.invitations.response_message_sid`.

Requirements:

- Multiple `NULL` values must remain allowed
- Non-NULL `response_message_sid` values must be unique
- Use the smallest, clearest Postgres-native approach
- Make the migration idempotent and safe for existing data on current `main`

If existing duplicate non-NULL values are discovered during migration authoring, stop and report rather than inventing silent cleanup behavior.

### 2) Transactional RPC

Add a Postgres RPC dedicated to invitation response processing.

The RPC must execute in one transaction and handle all required side effects for one inbound response:

- Validate target invitation row
- Enforce replay safety using `response_message_sid`
- Apply accept or pass transition only from a valid pending state
- Set `responded_at`
- Set `response_message_sid`
- Insert the required `learning_signals` row
- Update `users.invitation_backoff_count` according to response outcome
- Transition the relevant `conversation_sessions` row to `idle`
- Enqueue the required `sms_outbound_jobs` record
- Return a result payload sufficient for Ticket 3.1 runtime wiring

The RPC contract should be narrowly designed for the invitation-response use case only. Do not create a generic workflow engine.

## RPC Behavior Requirements

The RPC must:

- Be safe under duplicate Twilio webhook retries
- Guarantee no duplicated side effects for the same `response_message_sid`
- Reject invalid action values
- Reject terminal invitations that are already accepted, passed, or expired
- Preserve clear separation between:
  - successful first processing
  - duplicate replay
  - invalid/terminal-state attempt

The return shape should be minimal but explicit enough for the handler layer to decide what outbound behavior to take.

## Non-Goals

Do not:

- Implement `packages/messaging/src/handlers/handle-invitation-response.ts`
- Update router dispatch
- Update classifier logic
- Implement quorum logic
- Add broader invitation orchestration changes
- Change invitation taxonomy beyond what is required for this RPC
- Refactor unrelated linkup or post-event RPCs

## Schema / Naming Contract

Use the live schema on current `main`, not older prose wording:

- `invitation_type` uses current live values
- `time_window` remains `time_window`
- Any side-effect writes must target the actual current table/column names in the repo
