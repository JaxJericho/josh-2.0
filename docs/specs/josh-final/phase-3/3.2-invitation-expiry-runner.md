## Implementation

### File: `packages/core/src/invitation/expire-invitations.ts`

```ts
export async function expireStaleInvitations(): Promise<{
  expiredCount: number;
}>;
```

Steps:

1. Query invitations in batches of 50:

   ```sql
   SELECT id, user_id, invitation_type, activity_key, proposed_time_window
   FROM invitations
   WHERE state = 'pending'
     AND expires_at <= now()
   ORDER BY expires_at ASC
   LIMIT 50
   ```

2. For each expired invitation, inside a single transaction per invitation:
   - Update `invitations.state = 'expired'`
     - Guard: `WHERE id = $id AND state = 'pending'` — concurrent runner safety
   - Write to `learning_signals`:
     ```ts
     {
       user_id: invitation.user_id,
       signal_type: 'invitation_expired',
       payload: {
         invitation_id: invitation.id,
         invitation_type: invitation.invitation_type,
         activity_key: invitation.activity_key,
         proposed_time_window: invitation.proposed_time_window,
       },
     }
     ```
   - If `conversation_sessions.mode = 'awaiting_invitation_response'` for this user, set `mode = 'idle'`
   - Increment `users.invitation_backoff_count` by 1

3. Repeat until the batch returns zero rows.

4. Return `{ expiredCount }`.

#### Idempotency

The `WHERE state = 'pending'` guard ensures concurrent runners are safe. If two runners pick up the same row, the second update is a no-op (state is already `expired`). The learning signal write may duplicate — add an idempotency key to `learning_signals` if the table supports it; otherwise accept rare duplicates on this path (the signal is a soft input, not a financial record).

#### Error handling

Log and continue on individual invitation failure — do not abort the batch. Record the error with the `invitation_id` and `correlation_id`. Return the count of successfully expired invitations.

## Hard Rules for This Ticket

- Each invitation's state transition must use `WHERE state = 'pending'` as a guard — never update without this filter
- Batch size is 50 — do not load all stale invitations into memory
- Do not set `conversation_sessions.mode` unless it is currently `awaiting_invitation_response` — do not overwrite other active modes
- Process continues on individual invitation error — batch does not abort
- Correlation ID: generate one per runner execution, attach to all log events for that run
