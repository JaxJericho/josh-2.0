## Signal Definitions

### `interest_signatures`

What it captures: specific domains the user cares about beyond stated activity preferences. Inferred from conversational energy — what the user elaborates on spontaneously, returns to, or speaks about with specificity. Not the same as an activity pattern.

Examples: "urban infrastructure", "endurance sports", "cooking as craft", "live music discovery".

Shape:

```ts
type InterestSignature = {
  domain: string; // free-text label, e.g. "urban infrastructure"
  intensity: number; // float 0..1, inferred engagement level
  confidence: number; // float 0..1, extraction confidence
};

type InterestSignatures = InterestSignature[];
```

Confidence threshold for population: `>= 0.40`. Signatures below this threshold are not written.

Used by:

- Solo invitation selector (Phase 6, Ticket 6.2): alignment bonus = `0.15 × intensity`, capped at `0.20` total
- Cluster detector (Phase 6, Ticket 6.3): interest domain token overlap score per pair, contributes `interestOverlapScore × 0.10` to adjusted cluster ranking

### `relational_context`

What it captures: where the user is in their social life right now. Not a stable trait — a current-state snapshot.

Shape:

```ts
type RelationalContext = {
  life_stage_signal: string | null; // e.g. "new to city", "recently divorced", "empty nester"
  connection_motivation: string | null; // e.g. "rebuilding social circle", "boredom", "loneliness"
  social_history_hint: string | null; // e.g. "used to have a tight friend group but drifted"
};
```

All three fields are nullable within the object. The column itself is nullable. Population requires at least one field to be non-null.

Used by:

- Solo invitation selector (Phase 6, Ticket 6.2): category boost of `0.10` when `connection_motivation` keyword matches the activity's motive weights
