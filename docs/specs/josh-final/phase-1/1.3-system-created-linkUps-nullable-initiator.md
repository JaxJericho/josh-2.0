# Ticket 1.3 — `linkups` Schema Amendments: Nullable `initiator_user_id` + `system_created` (S)

## Context

This is the third ticket in Session 1. The existing `linkups.initiator_user_id` column is `NOT NULL`. Under the pivot model, system-generated group LinkUps have no human initiator — JOSH creates them directly. This ticket makes `initiator_user_id` nullable and adds a `system_created` boolean to distinguish system-created rows from user-initiated ones.

This is a schema-only change. No application logic changes in this ticket.

Read `AGENTS.md` and `Claude_Code_Instructions.md` before writing any code.

---

## Goal

Alter `linkups.initiator_user_id` to be nullable. Add `linkups.system_created boolean not null default false`. Update the `Linkup` TypeScript type and Zod schema to reflect the nullable column.

## Migration

File: `supabase/migrations/YYYYMMDD_linkups_nullable_initiator_system_created.sql`

```sql
-- Ticket 1.3: Allow system-created LinkUps (no human initiator).
-- initiator_user_id is nullable from this point forward.
-- system_created = true marks rows created by the invitation generator,
-- not by a user action.

begin;

-- Step 1: Drop the NOT NULL constraint on initiator_user_id.
-- The FK relationship (references public.users) is preserved.
-- ON DELETE RESTRICT is preserved — if initiator_user_id is set and that user
-- is deleted, the linkup row is protected from orphaning.
alter table public.linkups
  alter column initiator_user_id drop not null;

-- Step 2: Add system_created column.
-- All existing rows are user-initiated, so default = false is correct backfill.
alter table public.linkups
  add column system_created boolean not null default false;

-- Step 3: Add partial index for efficient queries on system-created LinkUps.
-- Used by the regional invitation generator and quorum runner.
create index linkups_system_created_idx on public.linkups(system_created, state)
  where system_created = true;

commit;
```

### Constraint Invariant

The application must enforce the following at write time (not in the DB — enforced in `cluster-detector.ts` and `dispatch-invitation.ts` in Phase 6):

- If `system_created = true`, then `initiator_user_id` must be `NULL`
- If `system_created = false`, then `initiator_user_id` must be non-null

This is a behavioral constraint, not a DB constraint. A DB-level check constraint is not added here because it would require a more complex expression and the application layer is the correct enforcement point for business logic of this kind.

## TypeScript Type Update

Locate the `Linkup` type in `packages/db/src/types/`. Update `initiator_user_id` from `string` to `string | null`:

```ts
// Before
export type Linkup = {
  // ...
  initiator_user_id: string;
  // ...
};

// After
export type Linkup = {
  // ...
  initiator_user_id: string | null;
  system_created: boolean;
  // ...
};
```

Update the corresponding Zod schema:

```ts
// Before
initiator_user_id: z.string().uuid(),

// After
initiator_user_id: z.string().uuid().nullable(),
system_created: z.boolean(),
```
