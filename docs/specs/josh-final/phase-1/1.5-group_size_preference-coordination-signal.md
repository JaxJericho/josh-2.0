# Ticket 1.5 — `group_size_preference` Coordination Signal on `profiles` (S)

## Context

This is the fifth ticket in Session 1. The quorum checker (Phase 6, Ticket 6.8) reads each acceptee's `group_size_preference` to determine whether a group's size satisfies all participants' stated preferences. This ticket adds the column to `profiles` and defines the shape of its value.

`group_size_preference` is not required for `complete_mvp`. It is populated by the holistic extractor when the user expresses a group size preference during the interview, and by `pass-pattern-detector.ts` (Phase 7, Ticket 7.2) if repeated pass signals indicate a preference change.

This is a schema-only change. No application logic changes in this ticket.

Read `AGENTS.md` and `Claude_Code_Instructions.md` before writing any code.

---

## Goal

Add `group_size_preference jsonb` to the `profiles` table. Define the `GroupSizePreference` TypeScript type. Update the `Profile` type to include the new field. No changes to `complete_mvp` threshold logic — that is enforced in `signal-coverage.ts` which already knows this field is not required.

## Migration

File: `supabase/migrations/YYYYMMDD_profiles_group_size_preference.sql`

```sql
-- Ticket 1.5: Add group_size_preference coordination signal to profiles.
-- This is a top-level column, distinct from profiles.preferences.group_size_pref.
-- group_size_preference stores the user's stated preference for group invitation size.
-- It is populated by the holistic extractor and by pass-pattern-detector.ts.
-- It is not required for complete_mvp.
-- Shape: { "min": integer, "max": integer }
-- Constraints: min >= 2, max <= 10, min <= max (enforced by application, not DB).

begin;

alter table public.profiles
  add column group_size_preference jsonb;

-- No index needed: this column is read per-user during quorum checks,
-- which are already keyed by user_id on an existing index.

commit;
```

### Value Shape

The value stored in `group_size_preference` is:

```json
{ "min": 2, "max": 6 }
```

Constraints enforced by the application (not the DB):

- `min` must be an integer >= 2
- `max` must be an integer <= 10
- `min` must be <= `max`

The quorum checker snapshot field `group_size_preference_snapshot` (stored in `linkup_invites` or equivalent — defined in Phase 6) copies this value at invitation dispatch time so that a user's preference change after dispatch does not affect an in-progress quorum check.

## TypeScript Type and Zod Schema

### New Type

Add `GroupSizePreference` to `packages/db/src/types/`:

```ts
export type GroupSizePreference = {
  min: number; // integer, >= 2
  max: number; // integer, <= 10
};
```

### Zod Schema

```ts
export const GroupSizePreferenceSchema = z
  .object({
    min: z.number().int().min(2),
    max: z.number().int().max(10),
  })
  .refine((data) => data.min <= data.max, {
    message: "min must be <= max",
  });
```
