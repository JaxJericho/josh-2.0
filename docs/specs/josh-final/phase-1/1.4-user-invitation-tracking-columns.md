# Ticket 1.4 — User Invitation Tracking Columns (S)

## Context

This is the fourth ticket in Session 1. The frequency guard (Phase 6, Ticket 6.1) enforces a 2-day cooldown, a 3-invitation weekly cap, and backoff suppression at `invitation_backoff_count >= 5`. It reads these values directly from the `users` row. This ticket adds the four columns the frequency guard requires.

This is a schema-only change. No application logic changes in this ticket.

## Migration

File: `supabase/migrations/YYYYMMDD_users_invitation_tracking_columns.sql`

```sql
-- Ticket 1.4: Add invitation tracking columns to users.
-- These columns are read by frequency-guard.ts (Phase 6, Ticket 6.1)
-- to enforce the 2-day cooldown, 3-invitation weekly cap,
-- and backoff suppression at invitation_backoff_count >= 5.

begin;

-- last_invited_at: timestamp of the most recent invitation dispatch.
-- NULL for users who have never received an invitation.
-- Used to enforce the 2-day minimum cooldown between invitations.
alter table public.users
  add column last_invited_at timestamptz;

-- invitation_week_start: start of the current rolling 7-day invitation window.
-- Set when the first invitation of a new week is dispatched.
-- Compared against now() to determine whether the window has rolled over.
-- NULL for users who have never received an invitation.
alter table public.users
  add column invitation_week_start timestamptz;

-- invitation_count_this_week: number of invitations dispatched within
-- the current rolling 7-day window.
-- Reset to 0 when invitation_week_start rolls over.
-- The weekly cap is 3 (enforced in frequency-guard.ts, not in the DB).
alter table public.users
  add column invitation_count_this_week int not null default 0;

-- invitation_backoff_count: cumulative count of consecutive passed or
-- expired invitations without an acceptance.
-- Incremented by the response handler and expiry runner.
-- Reset to 0 on any accepted invitation.
-- At count >= 5, the frequency guard suppresses new invitations until
-- re-engagement check-in clears the backoff (Phase 6, Ticket 6.6).
alter table public.users
  add column invitation_backoff_count int not null default 0;

-- Partial index for efficient frequency guard queries.
-- The guard only queries users who have been invited recently.
create index users_last_invited_idx on public.users(last_invited_at)
  where last_invited_at is not null;

commit;
```

### Semantic Rules (Enforced by Application, Not DB)

These rules are documented here for the Phase 6 implementer of `frequency-guard.ts`:

| Column                       | Reset condition                                                                            | Increment condition                                         |
| ---------------------------- | ------------------------------------------------------------------------------------------ | ----------------------------------------------------------- |
| `last_invited_at`            | Never reset; always overwritten on dispatch                                                | Set to `now()` on every dispatch                            |
| `invitation_week_start`      | Set to `now()` when rolling window expires (7 days since previous `invitation_week_start`) | Not incremented; replaced                                   |
| `invitation_count_this_week` | Set to `1` when window rolls over                                                          | Incremented by 1 on each dispatch within the current window |
| `invitation_backoff_count`   | Set to `0` on any accepted invitation                                                      | Incremented by 1 on each passed or expired invitation       |

The weekly cap is `3`. The backoff suppression threshold is `5`. Both are constants defined in `packages/core/src/invitation/constants.ts` (Phase 6). Do not hardcode them elsewhere.

## Hard Rules for This Ticket

- Do not add a DB check constraint enforcing the weekly cap or backoff threshold — these are application constants, not DB constraints
- All four columns are additive; no existing data is modified
- `invitation_count_this_week` and `invitation_backoff_count` must be `NOT NULL` with `DEFAULT 0` — they are never null in application logic
- `last_invited_at` and `invitation_week_start` are nullable because newly registered users have no invitation history
