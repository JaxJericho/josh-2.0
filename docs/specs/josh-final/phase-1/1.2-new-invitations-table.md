## Migration

File: `supabase/migrations/YYYYMMDD_add_invitations_table.sql`

Use today's date for `YYYYMMDD`.

```sql
-- Ticket 1.2: Create invitations table
-- Primary state machine entity for JOSH-initiated invitations (pivot model).
-- Covers both solo activity invitations and group LinkUp invitations.
-- Solo invitations have linkup_id = NULL.
-- Group invitations have linkup_id referencing a system-created linkups row.

begin;

create type public.invitation_state as enum (
  'pending',
  'accepted',
  'passed',
  'expired'
);

create type public.invitation_type as enum (
  'solo',
  'group'
);

create table public.invitations (
  id                    uuid primary key default gen_random_uuid(),
  user_id               uuid not null references public.users(id) on delete cascade,
  invitation_type       public.invitation_type not null,
  -- linkup_id is null for solo invitations, set for group invitations
  linkup_id             uuid references public.linkups(id) on delete set null,
  activity_key          text not null,
  time_window           text not null,
  state                 public.invitation_state not null default 'pending',
  expires_at            timestamptz not null,
  responded_at          timestamptz,
  -- response_message_sid enables idempotency on Twilio retries in the response handler
  response_message_sid  text,
  idempotency_key       text not null,
  correlation_id        uuid,
  created_at            timestamptz not null default now(),
  updated_at            timestamptz not null default now(),
  constraint invitations_idem_uniq unique (idempotency_key)
);

-- Primary query pattern: load pending invitations for a user
create index invitations_user_state_idx on public.invitations(user_id, state);

-- Primary query pattern: expire runner scans by expires_at
create index invitations_expires_idx on public.invitations(expires_at)
  where state = 'pending';

-- FK lookup for group invitation quorum check
create index invitations_linkup_idx on public.invitations(linkup_id)
  where linkup_id is not null;

-- updated_at trigger
create trigger invitations_updated_at
  before update on public.invitations
  for each row execute function public.set_updated_at();

-- RLS: users can read their own invitations only
alter table public.invitations enable row level security;

create policy invitations_select_own
  on public.invitations for select
  using (user_id = auth.uid());

commit;
```

### Idempotency Key Format

The idempotency key for each invitation is constructed by the application at dispatch time:

```
invitation:{userId}:{activityKey}:{isoWeek}
```

Where `isoWeek` is the ISO 8601 week string for the week the invitation was dispatched (e.g., `2026-W11`). This prevents duplicate invitations for the same user and activity within the same week while allowing re-invitation in subsequent weeks.

This key format is a constant. Define it in `packages/core/src/invitation/constants.ts` (created in Phase 6):

```ts
export const INVITATION_IDEMPOTENCY_KEY = (
  userId: string,
  activityKey: string,
  isoWeek: string,
): string => `invitation:${userId}:${activityKey}:${isoWeek}`;
```

Do not implement `constants.ts` in this ticket. Document the format here so the migration and type shape are consistent with how the key will be constructed.

### Expiry Window

`expires_at` is set by the application at dispatch time as `now() + interval '48 hours'`. This is a constant:

```ts
export const INVITATION_EXPIRY_HOURS = 48;
```

Again, define this in `constants.ts` during Phase 6, not now.

## Hard Rules for This Ticket

- Do not implement any application logic that reads from or writes to `invitations` — that is Phase 3 and Phase 6
- Do not add `invitation_state` or `invitation_type` to any existing enum — these are standalone new enums, not additions to `conversation_mode` or `learning_signal_type`
- `linkup_id` must be nullable in the migration; group invitations reference a system-created LinkUp but solo invitations have no associated LinkUp
- The idempotency key unique constraint is non-negotiable — it prevents duplicate invitations from concurrent dispatch runs
- RLS policy is read-only for users — no user-facing write policy; all writes are server-side via service role
