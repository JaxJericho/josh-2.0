## New Values

### `conversation_mode` — add one value

| Value                          | When active                                                                                                                                                                                                             |
| ------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `awaiting_invitation_response` | JOSH has dispatched an invitation (solo or group) and is waiting for the user's `accept` or `pass` reply. Set by `dispatch-invitation.ts` at dispatch time. Cleared by `handleInvitationResponse` or the expiry runner. |

This mode is distinct from `awaiting_invite_reply` (the 2.0 LinkUp invitation flow, which remains). Do not remove or modify `awaiting_invite_reply`.

### `learning_signal_type` — add three values

| Value                 | When written                                                                                                        |
| --------------------- | ------------------------------------------------------------------------------------------------------------------- |
| `invitation_accepted` | User replied "yes" or equivalent to a JOSH invitation. Written by `handleInvitationResponse` (Phase 3, Ticket 3.1). |
| `invitation_passed`   | User replied "pass", "no", or equivalent. Written by `handleInvitationResponse`.                                    |
| `invitation_expired`  | Invitation reached `expires_at` without a response. Written by the expiry runner (Phase 3, Ticket 3.2).             |

## Migration

File: `supabase/migrations/YYYYMMDD_add_pivot_enum_values.sql`

```sql
-- Ticket 1.6: Add pivot model enum values.
-- Additive only — no existing values are modified or removed.

begin;

-- Add awaiting_invitation_response to conversation_mode.
-- Distinct from awaiting_invite_reply (2.0 LinkUp flow, retained).
alter type public.conversation_mode
  add value if not exists 'awaiting_invitation_response';

-- Add invitation learning signal types.
alter type public.learning_signal_type
  add value if not exists 'invitation_accepted';

alter type public.learning_signal_type
  add value if not exists 'invitation_passed';

alter type public.learning_signal_type
  add value if not exists 'invitation_expired';

commit;
```

**Note on `add value if not exists`:** Postgres `ALTER TYPE ... ADD VALUE` is not transactional and cannot be rolled back within the same transaction. The `if not exists` guard makes the migration safe to re-run. This is the same pattern used in Phase 16.1.
