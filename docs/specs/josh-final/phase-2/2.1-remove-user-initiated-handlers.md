# Ticket 2.1 — Remove User-Initiated Handlers (M)

## Context

This is the first ticket in Session 2. The pivot model eliminates all user-initiated coordination paths. Users no longer express an intent that JOSH acts on — JOSH sends invitations and users respond. This ticket deletes the three handlers that implemented user-initiated paths, removes their dispatch entries from the router, and deletes the solo suggestion utility they depended on.

All Phase 1 schema migrations must be complete and applied before this ticket runs.

Read `AGENTS.md` and `Claude_Code_Instructions.md` before writing any code.

---

Delete four files. Remove their import and dispatch entries from the router. Confirm no other runtime file references the deleted code. Run typecheck.

## Step 2: Update the Conversation Router

File: `packages/messaging/conversation-router.ts`

Remove the following from this file:

**Imports to remove:**

```ts
import { handleOpenIntent } from "./handlers/handle-open-intent";
import { handleNamedPlanRequest } from "./handlers/handle-named-plan-request";
import { handlePlanSocialChoice } from "./handlers/handle-plan-social-choice";
```

**Dispatch entries to remove** (exact form may vary — remove whatever dispatches these three intents):

```ts
case 'OPEN_INTENT':
  return handleOpenIntent(userId, message, session);

case 'NAMED_PLAN_REQUEST':
  return handleNamedPlanRequest(userId, message, session);

case 'PLAN_SOCIAL_CHOICE':
  return handlePlanSocialChoice(userId, message, session);
```

**Fallback behavior for orphaned session modes:**

After removing the `PLAN_SOCIAL_CHOICE` dispatch entry, a session that is somehow in `awaiting_social_choice` or `pending_plan_confirmation` mode has no handler. This cannot happen with live users (there are none), but the router must not silently drop such a message. Replace each removed dispatch entry with a fallback that transitions the session to `idle` and sends a neutral message:

```ts
// OPEN_INTENT — user-initiated path retired under pivot model.
// Route to handleUnknown to surface a neutral response.
case 'OPEN_INTENT':
  return handleUnknown(userId, message, session);

// NAMED_PLAN_REQUEST — user-initiated path retired under pivot model.
case 'NAMED_PLAN_REQUEST':
  return handleUnknown(userId, message, session);

// PLAN_SOCIAL_CHOICE — awaiting_social_choice mode is orphaned under pivot model.
// Transition to idle and return unknown response.
case 'PLAN_SOCIAL_CHOICE':
  return handleUnknown(userId, message, session);
```

Do not remove the `PLAN_SOCIAL_CHOICE` case from the switch — route it to `handleUnknown` so any session stuck in `awaiting_social_choice` receives a response rather than a silent failure. The intent type itself is removed in Ticket 2.2.

**Do not touch:**

- `CONTACT_INVITE_RESPONSE` dispatch entry
- `POST_ACTIVITY_CHECKIN` dispatch entry
- `INVITE_RESPONSE` dispatch entry (2.0 LinkUp flow, retained)
- `INTERVIEW_ANSWER` dispatch entry
- `INTERVIEW_ANSWER_ABBREVIATED` dispatch entry
- `STOP/HELP/SYSTEM_COMMAND` handling
- `handleInviteResponse`, `handleInterviewAnswer`, `handleContactInviteResponse`, `handlePostActivityCheckin` — all retained

## Hard Rules for This Ticket

- Do not remove `awaiting_social_choice` or `pending_plan_confirmation` from the `conversation_mode` DB enum — that is a destructive migration deferred to a future cleanup session
- Do not remove `PLAN_SOCIAL_CHOICE`, `OPEN_INTENT`, or `NAMED_PLAN_REQUEST` from the TypeScript intent type union — that is Ticket 2.2
- Do not modify `handleInviteResponse` — the 2.0 LinkUp invitation flow is retained
- Do not touch `handleContactInviteResponse` — the member-to-member referral path is retained
- Route orphaned intent cases to `handleUnknown`, do not silently drop them
- Do not remove the `OPEN_INTENT` / `NAMED_PLAN_REQUEST` / `PLAN_SOCIAL_CHOICE` case labels from the router switch — redirect them; the labels are removed in Ticket 2.2 when the intent types are removed
