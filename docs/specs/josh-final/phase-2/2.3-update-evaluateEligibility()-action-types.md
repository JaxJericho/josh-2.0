# Ticket 2.3 — Update `evaluateEligibility()` Action Types (S)

## Context

This is the third and final ticket in Session 2. It runs after Tickets 2.1 and 2.2.

`evaluateEligibility()` was implemented in Phase 17.2 with three action types that matched the 3.0 spec at the time: `can_initiate_linkup`, `can_initiate_named_plan`, and `can_receive_contact_invitation`. Under the pivot model, users no longer initiate anything — JOSH initiates. Two of the three action types are now wrong, and a new one is needed to gate invitation dispatch.

This ticket updates the function signature, the runtime guard, the call sites, and the unit tests. No schema changes. No new tables.

Read `AGENTS.md` and `Claude_Code_Instructions.md` before writing any code.

---

## Goal

Replace `can_initiate_linkup` with `can_participate_in_linkup`. Remove `can_initiate_named_plan`. Add `can_receive_invitation`. Keep `can_receive_contact_invitation` unchanged. Update all call sites and the runtime guard.

---

## Action Type Changes

| Old                              | New                         | Reason                                                                                                                                                                                           |
| -------------------------------- | --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `can_initiate_linkup`            | `can_participate_in_linkup` | Users no longer create LinkUps. They accept invitations to system-created ones. The gate still checks subscription + open region — same conditions, new name reflecting the actual relationship. |
| `can_initiate_named_plan`        | removed                     | Handler deleted in Ticket 2.1. No remaining call sites.                                                                                                                                          |
| `can_receive_contact_invitation` | unchanged                   | Member-to-member referral path retained.                                                                                                                                                         |
| _(new)_                          | `can_receive_invitation`    | Gate checked by the invitation dispatch flow (Phase 6) before sending a JOSH-initiated invitation. Requires active subscription + profile state ≥ `complete_mvp`.                                |

---

## Gate Logic (New and Changed)

### `can_participate_in_linkup`

Conditions (unchanged from `can_initiate_linkup`):

- `region.status = 'open'`
- Active subscription (`subscriptions.status = 'active'`)

Checked by: `handleInvitationResponse` (Phase 3, Ticket 3.1) when a user accepts a group invitation, to confirm they are still eligible to participate before quorum is counted.

### `can_receive_invitation`

Conditions (new):

- Active subscription (`subscriptions.status = 'active'`)
- `profile.state` is `complete_mvp` or `complete_full`
- User is not on safety hold (`safety_hold_state = 'none'`)

Checked by: the regional invitation generator (Phase 6, Ticket 6.5) and `frequency-guard.ts` (Phase 6, Ticket 6.1) before dispatching any invitation. If this gate fails, the user is skipped silently — no SMS is sent.

### `can_receive_contact_invitation`

Conditions (unchanged):

- Any active user record exists (membership only, no subscription check)

Checked by: `handleContactInviteResponse`, unchanged.

### Update the gate implementations

**`can_participate_in_linkup`** (renamed from `can_initiate_linkup`):

```ts
case 'can_participate_in_linkup': {
  const region = await getRegionForUser(userId);
  const subscription = await getSubscription(userId);
  if (region.status !== 'open') {
    return { eligible: false, reason: 'INELIGIBLE_REGION' };
  }
  if (subscription.status !== 'active') {
    return { eligible: false, reason: 'INELIGIBLE_SUBSCRIPTION' };
  }
  return { eligible: true };
}
```

**`can_receive_invitation`** (new):

```ts
case 'can_receive_invitation': {
  const subscription = await getSubscription(userId);
  if (subscription.status !== 'active') {
    return { eligible: false, reason: 'INELIGIBLE_SUBSCRIPTION' };
  }
  const profile = await getProfile(userId);
  if (
    profile.state !== 'complete_mvp' &&
    profile.state !== 'complete_full'
  ) {
    return { eligible: false, reason: 'INELIGIBLE_PROFILE_INCOMPLETE' };
  }
  const safetyHold = await getSafetyHold(userId);
  if (safetyHold !== 'none') {
    return { eligible: false, reason: 'INELIGIBLE_SAFETY_HOLD' };
  }
  return { eligible: true };
}
```

**`can_receive_contact_invitation`** — implementation unchanged. Keep as-is.

**Remove `can_initiate_linkup` and `can_initiate_named_plan` cases entirely.**
