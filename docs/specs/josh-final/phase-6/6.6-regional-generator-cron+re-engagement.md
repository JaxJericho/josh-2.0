## Implementation

### File: `supabase/functions/scheduled/run-invitation-generator/index.ts`

Cron schedule: every 12 hours.

```ts
// Match the pattern of supabase/functions/scheduled/expire-invitations/index.ts exactly.
// QStash signature verification: required.

// 1. Load all regions where state = 'open'
const regions = await supabase.from("regions").select("id").eq("state", "open");

// 2. For each region, enqueue a QStash job targeting runRegionalGenerator(regionId)
//    Do not call runRegionalGenerator inline — enqueue only.
for (const region of regions.data) {
  await qstash.publishJSON({
    url: `${env.SUPABASE_FUNCTIONS_URL}/run-regional-generator`,
    body: { regionId: region.id },
  });
}

// 3. Log dispatch count
// { event: 'generator_cron.dispatched', regionCount: regions.data.length }
```

### File: `packages/core/src/invitation/re-engagement.ts`

```ts
export type ReEngagementResult =
  | { sent: true; userId: string }
  | { sent: false; reason: "safety_hold" | "threshold_not_met" };

export async function sendReEngagementMessage(
  userId: string,
): Promise<ReEngagementResult>;
```

Steps:

1. Load `users.invitation_backoff_count` and safety hold status.
2. If safety hold: return `{ sent: false, reason: 'safety_hold' }`.
3. If `invitation_backoff_count < INVITATION_BACKOFF_THRESHOLD`: return `{ sent: false, reason: 'threshold_not_met' }`.
4. Enqueue outbound SMS:
   > "Hey [first_name] — it's been a while. JOSH wants to make sure it's sending you the right things. What kinds of plans sound good to you lately?"
5. Set `conversation_sessions.mode = 'interviewing'` — this opens a lightweight profile refresh conversation.
6. Reset `users.invitation_backoff_count = 0`.
7. Return `{ sent: true, userId }`.

#### Where Re-engagement Fires

Re-engagement fires in the **cron function**, not in the regional generator. Before enqueueing a generator job for a region, the cron identifies users in that region with `invitation_backoff_count >= INVITATION_BACKOFF_THRESHOLD` and calls `sendReEngagementMessage` for each. These users are excluded from the generator run (they are not eligible via `checkInvitationEligibility` because the eligibility gate will block them, but the cron proactively handles them before the generator sees them).

Add to the cron function:

```ts
// Before enqueueing the generator job for each region:
// 1. Load users in this region with invitation_backoff_count >= INVITATION_BACKOFF_THRESHOLD
// 2. For each such user, call sendReEngagementMessage(userId)
// 3. Log re-engagement sends
```

## Hard Rules for This Ticket

- The cron enqueues QStash jobs per region — it does not call `runRegionalGenerator` inline. Inline calls block the cron and prevent parallel execution.
- Re-engagement resets `invitation_backoff_count = 0` atomically in the same operation as enqueuing the SMS
- Re-engagement must check safety hold before sending — never send an SMS to a user on safety hold
- `INVITATION_BACKOFF_THRESHOLD` imported from `constants.ts` — never hardcoded
