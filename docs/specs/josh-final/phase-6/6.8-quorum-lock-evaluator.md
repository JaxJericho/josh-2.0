## Implementation

### File: `packages/core/src/invitation/linkup-quorum.ts`

```ts
export type QuorumResult =
  | { locked: true; acceptedCount: number }
  | {
      locked: false;
      reason: "min_not_met" | "preferences_not_satisfied" | "still_pending";
    };

export async function evaluateLinkupQuorum(
  linkupId: string,
): Promise<QuorumResult>;
```

#### Steps

**1. Load all invitations for this linkup**

```sql
SELECT id, user_id, state, group_size_preference_snapshot, expires_at
FROM invitations
WHERE linkup_id = $linkupId
```

**2. Compute acceptance state**

```ts
const acceptedInvites = invitations.filter((i) => i.state === "accepted");
const pendingInvites = invitations.filter((i) => i.state === "pending");
const acceptedCount = acceptedInvites.length;
```

**3. Minimum check**

If `acceptedCount < 2`: return `{ locked: false, reason: 'min_not_met' }`.

**4. Preference satisfaction check**

For each accepted invitation, read `group_size_preference_snapshot`. Default `{ min: 2, max: 10 }` if null.

```ts
const prefSatisfied =
  acceptedCount >= Math.max(...acceptedInvites.map((i) => pref(i).min)) &&
  acceptedCount <= Math.min(...acceptedInvites.map((i) => pref(i).max));
```

- If `prefSatisfied` and `pendingInvites.length > 0`: return `{ locked: false, reason: 'still_pending' }`. More responses may satisfy a larger preference range.
- If `prefSatisfied` and `pendingInvites.length === 0`: proceed to lock.
- If `!prefSatisfied` and `pendingInvites.length > 0`: return `{ locked: false, reason: 'preferences_not_satisfied' }`.
- If `!prefSatisfied` and `pendingInvites.length === 0`: **conflict fallback** ŌĆö lock anyway if `acceptedCount >= 2`. Proceed to lock.

**5. Lock transaction**

Inside a single transaction:

- Update `linkups.state = 'locked'`, `locked_at = now()`, `acceptance_window_ends_at = now()`
  - Guard: `WHERE id = $linkupId AND state = 'broadcasting'` ŌĆö prevents double-lock under concurrent accepts
  - If this update returns 0 rows: the linkup was already locked (concurrent accept) ŌĆö return `{ locked: true, acceptedCount }` (the lock already happened)
- For each accepted user: insert `linkup_participants` row with `role = 'participant'`
  - Use `INSERT ... ON CONFLICT DO NOTHING` ŌĆö safe under concurrent calls
- For all remaining `pending` invitations: update `state = 'closed'`
- For all remaining `pending` linkup invites (if the 2.0 `linkup_invites` table is separate from `invitations`): close those too (grep first to confirm table structure)
- Enqueue confirmation SMS to each accepted user via `sms_outbound_jobs`:
  > "You're confirmed for [activity_display_name] with [acceptedCount - 1] other people [proposed_time_window]. JOSH will send a reminder closer to the time."

**6. Return**

Return `{ locked: true, acceptedCount }`.
