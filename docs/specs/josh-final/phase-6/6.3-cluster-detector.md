## Migration: `compatibility_score_cache` Table

File: `supabase/migrations/YYYYMMDD_add_score_cache.sql`

```sql
-- Ticket 6.3: Pair compatibility score cache for cluster detection.
-- Stores scored user pairs with a 24-hour TTL.
-- user_a_id < user_b_id enforced to ensure canonical ordering.

begin;

create table public.compatibility_score_cache (
  id             uuid primary key default gen_random_uuid(),
  user_a_id      uuid not null references public.users(id) on delete cascade,
  user_b_id      uuid not null references public.users(id) on delete cascade,
  profile_hash_a text not null,
  profile_hash_b text not null,
  score          numeric(4,3) not null,
  computed_at    timestamptz not null default now(),
  constraint score_cache_pair_uniq unique (user_a_id, user_b_id),
  check (user_a_id < user_b_id)  -- canonical ordering: lower UUID first
);

create index score_cache_computed_idx on public.compatibility_score_cache(computed_at);

commit;
```

Canonical ordering rule: when calling the scorer for pair (A, B), always set `user_a_id = min(A, B)` and `user_b_id = max(A, B)` by UUID string comparison. This ensures the unique constraint works correctly and cache lookups are unambiguous.

TTL: 24 hours. Cache entries older than `now() - interval '24 hours'` are stale and must be recomputed. Stale entries are not deleted — they are recomputed on next use and the row is updated.

## Implementation

### File: `packages/core/src/invitation/cluster-detector.ts`

```ts
import { CLUSTER_MIN_PAIRWISE_SCORE, MAX_CLUSTER_SIZE } from "./constants";

export type EligibleUser = {
  userId: string;
  profile: NormalizedProfile;
  groupSizePreference: { min: number; max: number }; // defaults { min: 2, max: 10 } if null
};

export type Cluster = {
  members: string[]; // userIds
  pairScores: Record<string, number>; // 'userA:userB' → score (canonical key order)
  clusterScore: number; // mean of all pair scores
  adjustedClusterScore: number; // clusterScore + (interestOverlapScore * 0.10)
  interestOverlapScore: number; // mean pairwise interest domain overlap
  activityKey: string;
  proposedTimeWindow: string;
};

export async function detectClusters(
  eligibleUsers: EligibleUser[],
): Promise<Cluster[]>;
```

#### Algorithm

```
1. Score all N×N eligible user pairs using the compatibility scorer.
   For each pair (A, B) where A ≠ B:
   - Canonical key: minId = min(A.userId, B.userId), maxId = max(A.userId, B.userId)
   - Check cache: query score_cache where user_a_id = minId AND user_b_id = maxId
     - If cache hit AND profile_hash_a matches current profile AND profile_hash_b matches current profile
       AND computed_at > now() - 24h: use cached score
     - Otherwise: call scorePair(A.profile, B.profile), write/update cache row

   - Also compute interestOverlapScore for the pair:
     interestOverlapScore = count of shared interest domain tokens /
       max(A.interest_signatures.length, B.interest_signatures.length, 1)
     Treat null interest_signatures as empty array. Token matching: lowercase split on
     spaces and hyphens. A "shared token" means the same token appears in at least one
     domain from each user's signatures array.

2. Sort pairs descending by score.

3. Initialize: clustered = new Set<string>(), clusters = []

4. For each pair [A, B] in sorted order:
   a. Skip if A or B is already in clustered
   b. Skip if score < CLUSTER_MIN_PAIRWISE_SCORE
   c. Seed candidate cluster: members = [A, B]
   d. For each remaining unclustered user C, sorted by their mean score against all current
      cluster members (descending):
      - Compute C's compatibility score against every current cluster member
      - If min(all those scores) >= CLUSTER_MIN_PAIRWISE_SCORE: add C to candidate cluster
      - If cluster.members.length reaches MAX_CLUSTER_SIZE: stop adding
   e. Finalize: add all members to clustered; compute cluster metrics:
      - clusterScore = mean of all pair scores within the cluster
      - interestOverlapScore = mean of all pair interestOverlapScores within the cluster
      - adjustedClusterScore = clusterScore + (interestOverlapScore * 0.10)
   f. Push cluster to output

5. Return clusters sorted descending by adjustedClusterScore.
   (Users not in any cluster are unclustered — caller handles solo fallback.)
```

#### Activity Selection for a Cluster

For each valid cluster:

1. Union all members' activity patterns — find activities that pass boundary filters (`no_thanks[]`) for all members
2. Score each candidate activity: mean motive overlap across all members (same formula as solo selector)
3. Select highest-scoring activity
4. If no activity passes all members' boundary filters: mark cluster `{ activityKey: null, activityUnresolvable: true }` — caller skips this cluster and falls back to solo for all members

#### Time Window Selection for a Cluster

Intersect all members' `scheduling_availability` buckets. Select the highest-confidence overlapping window. If no overlap exists: mark cluster as unresolvable for time window — caller falls back to solo.

  - Floor enforcement: user F with score 0.60 against B is not added to [A,B,C] cluster

  - Activity unresolvable: cluster where no activity clears all members' boundaries → marked unresolvable
  - Time window unresolvable: no scheduling overlap → marked unresolvable
  - Cache hit: pair score returned from cache without scorer call
  - Cache miss/stale: scorer called, cache updated
  - Interest overlap: two users with matching domain token → overlap score > 0
  - Adjusted score ranking: cluster with higher interestOverlapScore ranks above cluster with same clusterScore

## Hard Rules for This Ticket

- `CLUSTER_MIN_PAIRWISE_SCORE` and `MAX_CLUSTER_SIZE` must be imported from `constants.ts` — never hardcoded

- Unit tests must use fixed scores, not random inputs — deterministic outcomes required
