## Implementation

### File: `packages/core/src/invitation/solo-invitation-selector.ts`

```ts
export type SoloInvitationCandidate = {
  activityKey: string;
  proposedTimeWindow: string;
  locationHint: string | null;
  selectionScore: number;
  explainability: {
    motiveScore: number;
    diversityBonus: number;
    noveltyBonus: number;
    interestAlignmentBonus: number;
    relationalContextBonus: number;
  };
};

export async function selectSoloInvitation(
  userId: string,
): Promise<SoloInvitationCandidate | null>;
```

#### Steps

**1. Load profile signals**

Load from `profiles` for the user:

- `coordination_dimensions`
- `activity_patterns` (including `motive_weights` per activity)
- `scheduling_availability`
- `notice_preference`
- `boundaries` (including `no_thanks[]`)
- `interest_signatures` (may be null — treat as empty array)
- `relational_context` (may be null)

**2. Load activity catalog**

Load all `activity_catalog` entries where `regional_availability` matches the user's region.

**3. Filter**

Remove activities that fail any of these filters:

- Activity key is in `profiles.boundaries.no_thanks[]`
- Activity was invited in the past 14 days: query `invitations` where `user_id = userId AND activity_key = ? AND offered_at > now() - interval '14 days'`
- Activity's `preferred_windows` have no overlap with the user's `scheduling_availability` buckets

**4. Score remaining activities**

For each remaining activity, compute a composite score:

_Base motive score_: Using the same formula from the compatibility scorer:

```
motiveScore = Σ over motives min(activity.motive_weight[motive], profile.activity_patterns.motive_weights[motive] ?? 0)
```

_Diversity bonus_ (`+0.10`): Activity category is not present in the user's last 3 accepted `invitations`. Query `invitations` where `user_id = userId AND state = 'accepted' ORDER BY responded_at DESC LIMIT 3`.

_Novelty bonus_ (`+0.05`): Activity key is not in the user's `activity_patterns` entries (user has no history with this activity).

_Interest alignment bonus_ (depth signal — may be 0 if `interest_signatures` is null):

```
interestAlignmentBonus = min(0.20,
  Σ over interest_signatures where domain overlaps activity.interest_tags:
    0.15 × signature.intensity
)
```

Domain overlap check: tokenize `signature.domain` and `activity.interest_tags` — overlap if any token matches. If `interest_signatures` is null: bonus = 0.

_Relational context bonus_ (`+0.10` — depth signal, may be 0 if `relational_context` is null):
If `relational_context.connection_motivation` is not null, check if any keyword from `connection_motivation` matches the activity's motive tags. If match: bonus = 0.10. If `relational_context` is null: bonus = 0.

_Composite score_:

```
score = motiveScore + diversityBonus + noveltyBonus + interestAlignmentBonus + relationalContextBonus
```

**5. Select activity**

Select the highest-scoring activity. If all activities score 0 (no motive overlap and no bonuses), select the highest scoring by motive alone — do not return null purely on low score.

Return null only if:

- No activities remain after filtering (all excluded by boundaries, recency, or availability)
- The user's profile has no `scheduling_availability` entries at all (cannot select a time window)

**6. Select time window**

Intersect `activity_catalog.preferred_windows` for the selected activity with the user's `scheduling_availability` buckets. Select the bucket with the highest confidence value in the user's profile. If multiple windows are tied, prefer the soonest in the week (Sunday < Monday < ... < Saturday).
