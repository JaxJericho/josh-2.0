## Implementation

### File: `packages/core/src/invitation/dispatch-single-user-invitation.ts`

```ts
export async function dispatchSingleUserInvitation(
  userId: string,
): Promise<void>;
```

Steps:

1. Call `selectSoloInvitation(userId)`.
2. If candidate returned: call `dispatchInvitation({ userId, invitationType: 'solo', activityKey, proposedTimeWindow, locationHint, correlationId: generateCorrelationId() })`.
3. If null returned: log `{ event: 'cold_start.no_activity_available', userId }`. This is not an error — it is a data gap (user's profile doesn't produce an eligible activity at this moment).

### File: `packages/core/src/invitation/cold-start-trigger.ts`

```ts
export async function enqueueColdStartInvitation(userId: string): Promise<void>;
```

Steps:

1. **Idempotency check**: Query `invitations` where `user_id = userId AND offered_at > now() - interval '24 hours'`. If any row exists: log and return — do not enqueue a second job.
2. **Compute delay**: `delayMs = randomInt(1 * 60 * 60 * 1000, 23 * 60 * 60 * 1000)` (1–23 hours in milliseconds).
3. **Enqueue QStash job**: Target `dispatchSingleUserInvitation(userId)` with `delayMs`. Use the QStash `delay` parameter (milliseconds).

### Integration Point in `profile-writer.ts`

Find the location in `profile-writer.ts` where the profile write completes and `getSignalCoverageStatus` is evaluated. Add transition detection:

```ts
// Before the write: read current profile_state
const previousProfileState = existingProfile?.profile_state ?? null;

// ... existing write logic ...

// After the write: check for mvpComplete transition
const newCoverage = getSignalCoverageStatus(updatedProfile);
const justReachedMvpComplete =
  newCoverage.mvpComplete === true &&
  previousProfileState !== "complete_mvp" &&
  previousProfileState !== "complete_full";

if (justReachedMvpComplete) {
  await enqueueColdStartInvitation(userId);
}
```

This fires once: on the first write where `mvpComplete` becomes true. Subsequent writes to an already-`complete_mvp` profile do not re-trigger because `previousProfileState` is now `complete_mvp` or `complete_full`.

## Hard Rules for This Ticket

- The transition detection must compare the profile state before and after the write — not just the new state. A write that keeps the profile at `complete_mvp` must not re-trigger.
- The idempotency check in `cold-start-trigger.ts` prevents double-enqueueing if `profile-writer.ts` is called multiple times rapidly (e.g., due to webhook retries)
- `dispatchSingleUserInvitation` logs a data gap on null selector result — never throws
- Run `pnpm typecheck` immediately after the `profile-writer.ts` change. Any type errors are this ticket's responsibility.
