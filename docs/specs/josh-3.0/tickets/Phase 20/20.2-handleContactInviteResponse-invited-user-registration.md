# Ticket 20.2 — handleContactInviteResponse: Invited User Registration

## Role

Principal Systems Architect & Full-Stack Engineer (routing correctness, replay safety, production-safe registration)

## Phase

Phase 20 — Invited User Flow

## Complexity

High

## What This Ticket Builds

A deterministic inbound SMS handler that activates when a non-member (unknown phone number) replies to an invitation SMS and we have a pending `contact_invitations` record for their `from_phone_hash`.

This handler:

- Creates the user + profile + session in an “invited abbreviated interview” mode
- Marks the invitation accepted
- Sends the exact invited welcome message
- Is safe under retries, duplicates, and race conditions
- Guarantees the invited completion path uses `complete_invited` (not `complete_mvp`)

## Why It Matters

This is the front door for invited users. If it’s flaky, we either:

- Create duplicate users/sessions, or
- Drop invited users into the wrong onboarding flow, or
- Lose the invitation linkage and cannot attribute or coordinate the new relationship.

We need a deterministic, testable, replay-safe invited registration path that integrates with Phase 19’s intent-first router and Phase 16’s invitation lookup priority.

## Behavioral Constraints

### Additive vs Destructive

- Additive only.
- Do not refactor unrelated onboarding, session, or router logic.
- Only make changes required to wire/implement this handler.

### Replay Safety / Idempotency

- Safe under:
  - Twilio retries / duplicate webhooks
  - QStash/worker retries (if this is invoked asynchronously)
  - User double-sends (e.g., “YES” twice)
- Idempotency must be enforced primarily via DB uniqueness constraints + “get-or-create” semantics.

### Routing Determinism

- If `contact_invitations` exists for `from_phone_hash` and is in a pending/active state, this handler must be selected deterministically before any generic onboarding or open-intent handling.
- The handler must not depend on LLM classification.

### Completion State Guarantee

- Invited onboarding must never mark the profile as `complete_mvp`.
- This ticket must ensure the invited registration flow sets up the abbreviated interview mode and target completion state that results in `complete_invited`.

### Do Not Change

- Do not send outbound “cold invite” SMS in this ticket (Ticket 20.1 creates a gated outbound job; dispatch is gated).
- Do not implement abbreviated interview question/answer processing in this ticket (Ticket 20.3).
- Do not change Phase 19 social choice flow behavior.

## Schema Changes (If Applicable)

Prefer no schema changes if Phase 16 already includes required types/constraints.

If missing, add additive, replay-safe constraints to guarantee idempotency:

- Users table has a unique constraint on normalized phone hash (or equivalent unique identifier).
- Conversation sessions table has uniqueness that prevents multiple “active invited onboarding sessions” for the same user/phone at the same time (or enforce via logic + a stable query selecting the newest active session).
- Contact invitations table supports:
  - A status transition to “accepted”
  - A link to the newly created user/profile/session (if modeled), or at least storing accepted timestamps and accepted user id.

All migrations must be replay-safe and additive only.

## Runtime Changes

### Trigger Condition (Eligibility For This Handler)

Input: inbound SMS payload (already normalized by router) includes:

- `from_phone_hash`
- `from_phone_e164` (or equivalent; never store raw in logs/audit)
- `body_text`
- Router context fields (existing)

The handler activates when:

1. The sender is not a known user OR the router’s “pending invitation short-circuit” is triggered for this phone hash, AND
2. A `contact_invitations` row exists where:
   - `invitee_phone_hash == from_phone_hash`
   - status indicates pending/active (not accepted/expired/declined)

If multiple pending invitations exist (should not happen if 20.1 constraints exist), pick the most recent created_at deterministically.

### Accept/Decline Parsing

This handler is specifically for invitation response. It must interpret the inbound message body as one of:

- Accept intent: “yes”, “y”, “ok”, “sure”, “accept”, “join” (case/whitespace-insensitive)
- Decline intent: “no”, “n”, “stop”, “decline” (case/whitespace-insensitive)

Rules:

- STOP always wins (system command behavior remains globally enforced).
- If ambiguous or not recognized, reply with a single clarification prompt that asks for YES/NO (deterministic text; do not invoke LLM).

### Core Behavior: Accept Path

On accept:

1. Mark invitation accepted in a replay-safe way:
   - Update invitation status to accepted if currently pending.
   - If already accepted, proceed idempotently (do not error).
2. Create or reuse the invited user:
   - Registration source must be `contact_invitation`.
   - Enforce uniqueness on phone hash.
   - If concurrent creation occurs, handle the unique-constraint violation by selecting the existing user and continuing.
3. Create or reuse an invited profile:
   - Create a profile for the user if none exists.
   - Profile state must begin as incomplete and targeted for invited abbreviated completion (do not set `complete_mvp` anywhere).
4. Create or reuse a conversation session:
   - Create a session with:
     - `mode = interviewing_abbreviated` (exact enum/value as defined in 3.0)
     - Any fields required by the abbreviated interview engine (e.g., state_token initialization, turn counters, etc.)
   - If an active invited abbreviated session already exists, reuse it.
5. Link records:
   - Store linkages needed for audit/traceability:
     - invitation.accepted_user_id (if available)
     - invitation.accepted_profile_id (if available)
     - invitation.accepted_session_id (if available)
   - If schema doesn’t support direct links, ensure audit events include the ids so we can correlate.
6. Send the invited welcome message (verbatim):
   - Use the canonical message string defined in the ticket spec file in the repo.
   - Message must be sent once; if replayed, do not double-send (idempotency key must exist, e.g., by session + message purpose).

### Core Behavior: Decline Path

On decline:

1. Mark invitation declined (or equivalent terminal status) in a replay-safe way.
2. Send a single confirmation message acknowledging decline (verbatim from spec).
3. Do not create a user/profile/session.

### Unknown/Unrecognized Response Path

If response is neither accept nor decline:

- Send a deterministic clarification message:
  - “Reply YES to join, or NO to decline.” (use spec’s exact copy if provided)
- Do not change invitation status.
- Do not create user/profile/session.

### Auditing

Emit structured audit events for each outcome:

- `contact_invite_response_received`
- `contact_invite_accepted` or `contact_invite_declined` or `contact_invite_clarify_prompted`

Audit payload must include:

- invitation_id
- inviter_user_id (from invitation)
- invitee_phone_hash (never raw phone)
- created/reused flags for user/profile/session
- session mode set
- idempotency status (first-run vs replay)

## Dependencies

- Ticket 20.1 exists and enforces sane invitation uniqueness semantics.
- Phase 19 router provides early “pending invitation” short-circuiting by phone hash.
- Shared hashing utilities and user lookup conventions exist.
- Messaging send utilities support idempotency keys (or we add minimal idempotency support for this handler only).
