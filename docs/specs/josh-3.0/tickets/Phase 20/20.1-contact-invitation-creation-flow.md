# Ticket 20.1 — Contact Invitation Creation Flow

## Role

Principal Systems Architect & Full-Stack Engineer (production safety + replay-safe messaging)

## Phase

Phase 20 — Invited User Flow

## Complexity

Medium

## What This Ticket Builds

A deterministic, replay-safe backend flow that lets an existing JOSH member create a contact invitation for a non-member by phone number, and prepares (but does not dispatch) the outbound invite SMS via an outbound job record.

This ticket creates the invitation record and the “ready to send” outbound job record. It does not send an SMS to a non-member.

## Why It Matters

Phase 20 depends on being able to invite a non-member into the “invited onboarding” path in a way that is:

- Safe to retry (no duplicate invites / jobs)
- Auditable (we can prove what happened)
- Compliant with the current operational constraint (do not actually dispatch cold outbound SMS until A2P is cleared)

## Behavioral Constraints

### Additive vs Destructive

- Additive only. Do not remove or refactor unrelated flows.
- Do not change Phase 19 intent routing behavior except to add the minimal new wiring required for “invite creation” to be reachable.

### Replay Safety / Idempotency

- The invitation creation path must be safe under retries, double submits, webhook replays, or worker retries.
- Idempotency must be enforced at the DB boundary, not only in application logic.

### Compliance Constraint (A2P Gate)

- This ticket must NOT send a Twilio SMS to a non-member.
- It may create an outbound job row that represents “to be dispatched later” once A2P is cleared.

### Do Not Change

- Do not modify the invited user acceptance/registration flow (Ticket 20.2).
- Do not modify the abbreviated interview completion logic (Ticket 20.3).
- Do not change baseline guardrail tests other than adding new tests specific to this ticket.

## Schema Changes (If Applicable)

Prefer **no schema changes** if the Phase 16 schema already contains what is needed.

If any of these do not exist, add an **additive, replay-safe migration**:

- `contact_invitations` (or equivalent) table with a uniqueness constraint that prevents duplicates for the same `(inviter_user_id, invitee_phone_hash, status in pending/sent)` or equivalent.
- `sms_outbound_jobs` (or equivalent) table for queued outbound messages.

Minimum required DB invariants for idempotency:

- A unique constraint that prevents creating more than one “active/pending” invitation for the same inviter → invitee pair.
- A unique constraint that prevents multiple outbound jobs for the same invitation and message purpose.

All migrations must:

- Use `if not exists` / `create ... if not exists` patterns where possible
- Be replay-safe
- Avoid destructive edits
- Avoid seed data in migrations

## Runtime Changes

### Entry Point (How The Flow Is Triggered)

Implement a server-side entry point that can be called by the conversation layer when a member requests to invite someone. Accept a minimal payload:

- `inviter_user_id` (existing member)
- `inviter_profile_id` (existing member profile)
- `invitee_phone_e164` (required)
- `invitee_display_name` (optional)
- `invitation_context` (optional: string blob for audit/debug; not required for behavior)

Phone normalization:

- Normalize to E.164 before hashing/storage. Reject if invalid.

### Core Behavior

When the member triggers an invite:

1. Compute `invitee_phone_hash` using the canonical hashing function used elsewhere in routing (same hashing strategy as Phase 19/20 invite lookup).
2. Upsert/ensure the inviter’s contact circle membership exists (if Phase 16 defines a “contact circle” or equivalent relationship table).
   - If there is no contact circle concept in schema, skip this step (do not invent a new model in this ticket).
3. Create or reuse a `contact_invitations` row in a **replay-safe** way:
   - If an active/pending invitation already exists for this inviter + invitee hash, return it (do not create a new one).
   - Status should be an “active/pending invite awaiting response” state (use existing enum/state if present; otherwise define a stable string value and centralize it).
4. Create or reuse an outbound job row in `sms_outbound_jobs` (or equivalent):
   - The job represents “send invite SMS to invitee,” but MUST be marked as **not dispatchable** until A2P is cleared.
   - Store enough payload for later dispatch: destination phone, template/message key, and metadata linking to the invitation id.
   - The job must be idempotent: one job per invitation per purpose.

### What The Invite SMS Should Contain (Stored For Later Dispatch)

This ticket should not perfect copywriting. It must store a deterministic template key/message type and the minimal variables required for Ticket 20.2 and ops/debugging:

- inviter display name (or fallback)
- an instruction for the invitee to reply (e.g., “Reply YES to join”)
- a short statement that this is an invitation from a friend

Do not include links.

### Auditing

Emit a structured audit event (existing audit system) with:

- inviter identifiers (user_id, phone hash if available)
- invitee phone hash (never raw phone in logs)
- invitation id
- outbound job id
- outcome: created vs reused
- reason: “contact_invitation_created”

## Dependencies

- Phase 19 intent-first router is merged and stable.
- Phase 16 database tables and hashing utilities exist (or are added via additive migrations as above).
- Existing outbound jobs / messaging queue infrastructure exists (even if dispatch is gated).

## Deliverables

- A callable server-side function / handler that executes the invite creation flow.
- DB-level idempotency constraints (only if missing).
- Unit tests for idempotency and behavior (create vs reuse).
- Router/handler wiring so the flow is reachable from the conversation layer (minimal scope).

## Verification Checklist

### Local

- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`
- `pnpm build`

### Behavioral Tests (Must Add)

- Creating a new invite produces exactly:
  - 1 invitation row
  - 1 outbound job row (gated / non-dispatchable)
- Replaying the same invite request (same inviter + invitee) does not create duplicates.
- Creating an invite for a different invitee creates a separate invitation/job.
- Raw phone numbers never appear in logs/audit payloads (hash only).

### Staging (If Available)

- `supabase db push --linked`
- SQL spot checks (exact table names may differ; adapt to actual schema):
  - Verify invitation row exists for inviter + invitee hash
  - Verify outbound job row exists and is flagged non-dispatchable / gated
  - Verify uniqueness constraints prevent duplicates

## Completion Criteria

- A member can trigger “invite creation” and the system:
  - Creates or reuses a single active invitation (idempotent)
  - Creates or reuses a single outbound job record (idempotent)
  - Does not send an SMS to the invitee
  - Emits a structured audit event with hashes only
- All verification gates pass (allowing only the known baseline `pnpm test` failure on main, if still present).
