# Ticket 20.3 — handleInterviewAnswerAbbreviated: Abbreviated Interview Engine

## Role

Principal Systems Architect & Full-Stack Engineer (state machines, LLM extraction correctness, deterministic completion)

## Phase

Phase 20 — Invited User Flow

## Complexity

High

## What This Ticket Builds

A new interview handler and state logic for the “invited abbreviated interview” mode. It processes inbound interview answers, performs extraction/scoring into the 3.0 coordination model, and completes the invited onboarding using a reduced threshold.

This ticket must:

- Implement `handleInterviewAnswerAbbreviated` (or equivalent) for session mode `interviewing_abbreviated`
- Persist extracted signals/dimensions to the profile
- Determine completion using invited thresholds
- Transition the profile to `complete_invited` (never `complete_mvp`)
- Send the invited wrap message verbatim
- Preserve replay safety and deterministic behavior

## Why It Matters

Invited users need a shorter, lower-friction onboarding that still produces enough signal quality to drive LinkUp and plan behaviors. The main interview flow is too heavy for invites. The abbreviated engine is the “fast lane” that still lands in a valid 3.0 profile state.

## Behavioral Constraints

### Additive vs Destructive

- Additive only.
- Do not refactor or change the standard interview handler behavior.
- Do not change the extraction model contracts except to reuse them or add the minimal invited-specific wrapper.

### Replay Safety / Idempotency

- Safe under:
  - Duplicate inbound messages
  - Worker retries
  - Partial failures between extraction and persistence
- Must prevent duplicate “completion” side effects:
  - No double wrap message
  - No flip-flopping profile states

### Deterministic Completion Rules

Completion for invited abbreviated onboarding is **strictly**:

- Coordination Dimensions:
  - Any **3 of 6** dimensions with confidence **≥ 0.55**
- Coordination Signals:
  - Any **1 of 3** signals with confidence **≥ 0.60**

When both conditions are satisfied:

- Set `profile_state = complete_invited`
- End/transition session out of `interviewing_abbreviated` (exact next mode per existing conventions; commonly `idle`)
- Send the invited wrap message once (verbatim from spec)

### Do Not Change

- Do not change how full interview completion works (`complete_mvp` path).
- Do not change Phase 19 intent routing.
- Do not change contact invitation acceptance/registration (Ticket 20.2) other than any minimal session mode wiring already expected.

## Schema Changes (If Applicable)

Prefer no schema changes if Phase 16/18 already created:

- `coordination_dimensions` storage (6 dimensions)
- `coordination_signals` storage (3 signals)
- Per-field confidence storage
- `profile_state` values including `complete_invited`

If `complete_invited` is not already part of the profile state enum/type, add it as an additive migration/type update (replay-safe).

No destructive migrations.

## Runtime Changes

### Trigger Condition

This handler runs when:

- The active `conversation_sessions.mode` is `interviewing_abbreviated`, AND
- An inbound message is classified as an “interview answer” for the abbreviated flow (this should not require LLM intent classification; mode-based dispatch is sufficient)

### Core Handler Responsibilities

On each inbound message while in `interviewing_abbreviated`:

1. Load the active session + associated profile.
2. Extract signals/dimensions from the message using the existing 3.0 extraction pipeline:
   - Must reuse the canonical extraction schemas/types introduced in Phases 17/18 (holistic 6-dimension model).
   - Do not introduce a second competing extraction format.
3. Persist extracted values to the profile in a merge-safe way:
   - Update only the fields with new or higher-confidence values.
   - Never lower confidence unless explicitly defined in the existing adaptation system.
4. Evaluate invited completion criteria (3-of-6, 1-of-3 using thresholds).
5. If complete:
   - Atomically set profile_state to `complete_invited` (only if not already complete)
   - Transition session out of `interviewing_abbreviated` to the next stable mode (typically `idle` unless the 3.0 state machine dictates otherwise)
   - Send the invited wrap message exactly once using an idempotency key scoped to (profile_id, purpose) or (session_id, purpose)
6. If not complete:
   - Continue the abbreviated interview:
     - Either ask the next abbreviated question deterministically, or
     - If Phase 16.5 dropout recovery handles prompting, follow that approach and keep handler minimal.
   - The handler must not loop or spam; one inbound message produces at most one outbound message.

### Question Selection Strategy (Deterministic)

The abbreviated engine must prefer gathering missing required signals:

- Prioritize dimensions/signals not yet above threshold.
- If multiple are missing, pick in a deterministic order (stable ordering by key).

Do not use the LLM to decide what to ask next beyond the existing extraction call.

### Dropout Recovery

Dropout recovery for invited abbreviated mode should use the already-defined recovery mechanism (Phase 16.5 note in build plan). This ticket should:

- Ensure abbreviated mode is compatible with the existing recovery logic, or
- Add only the minimal mode inclusion needed so recovery works.

Do not design a new recovery system.

### Auditing

Emit structured audit events for:

- `interview_abbreviated_answer_processed`
- `interview_abbreviated_progress_updated`
- `interview_abbreviated_completed`

Audit payload must include:

- profile_id, session_id
- which dimensions/signals were updated (keys only; no raw user text)
- completion evaluation snapshot (counts above thresholds)
- idempotency info (first-run vs replay)
