# Ticket 21.4 — Remove Legacy Fingerprint Compatibility Pipeline (L)

## Role

Principal Systems Architect & Full-Stack Engineer (dead-code removal, schema cleanup)

## Phase

Phase 21 — Matching System Updates, Destructive Migrations, And Legacy Removal

## Complexity

Large

## What This Ticket Builds

Removes the legacy 2.0 fingerprint compatibility pipeline by deleting dead code and dropping legacy tables **only after confirming they are unused** by any runtime path.

Targets:

- Delete legacy writer: `packages/core/compatibility-signal-writer.ts`
- Drop legacy tables (if unused):
  - `profile_compatibility_signals`
  - `profile_compatibility_scores`

## Scope Locks (Anti-Scope-Creep)

### Allowed Change Surface

- `packages/core/compatibility-signal-writer.ts` and its direct call sites (if any)
- Supabase migrations that drop the two legacy tables (if confirmed unused)
- Tests that fail due to removals (only minimal updates)

### Forbidden

- No new compatibility pipeline
- No changes to 3.0 scoring (that’s Ticket 21.3)
- No broad refactors
- No table renames or reshaping other models

### No New Concepts Rule

If any consumer exists, do not invent new pipelines. Replace consumer with existing 3.0 mechanisms or remove the consumer.

### Production Ready Definition

- Zero references to the legacy writer remain
- DB drops apply cleanly (staging verified)
- App still boots; matching still runs
- `pnpm lint`, `pnpm typecheck`, `pnpm build` pass; `pnpm test` passes except known baseline guardrail (if still present)

## Pre-Removal Checklist (Must Be Verified Before Deleting)

- Grep for imports/call sites of `compatibility-signal-writer` in:
  - `supabase/functions`
  - `packages/*`
  - `scripts/*`
- Confirm the two tables are not read/written in runtime paths:
  - grep SQL and query builders for `profile_compatibility_signals` and `profile_compatibility_scores`

If any consumer exists:

- Do not merge until consumer is removed or replaced within the same ticket.

## Requirements (Exact)

- If confirmed unused:
  - Delete `packages/core/compatibility-signal-writer.ts`
  - Add a migration to drop both tables
- Migration must be safe to run on staging first; include verification notes in PR:
  - `supabase db push --linked`
  - confirm tables dropped
  - run minimal matching job smoke (existing script) without regressions

## Deliverables

- Legacy writer removed (and no remaining references)
- Migration: `supabase/migrations/YYYYMMDD_drop_legacy_compatibility_tables.sql`
- Staging verification notes included in PR description

## Verification Checklist

### Local

- `supabase db reset --no-seed`
- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`
- `pnpm build`

### Staging

- `supabase db push --linked`
- Verify both tables are gone
- Confirm app still runs and matching still runs

## Completion Criteria

- Legacy writer deleted and unreferenced
- Legacy tables dropped via migration (if unused)
- Verified in staging
