# Ticket 21.5 — Destructive Enum Migrations (M)

## Role

Principal Systems Architect & Full-Stack Engineer (database migrations, zero-downtime safety)

## Phase

Phase 21 — Matching System Updates, Destructive Migrations, And Legacy Removal

## Complexity

Medium

## What This Ticket Builds

Destructive enum recreation migrations to remove deprecated enum values from:

- `learning_signal_type`:
  - remove `match_preview_accepted`, `match_preview_rejected`, `match_preview_expired`
- `match_mode`:
  - remove `one_to_one` (leave `linkup` only)

Includes rollback documentation and mandatory staging verification queries.

## Scope Locks (Anti-Scope-Creep)

### Allowed Change Surface

- Supabase migration(s) required to recreate enums
- TypeScript types/constants that reference these enums
- A rollback runbook doc
- Minimal tests that fail due to enum/type removal

### Forbidden

- No changes to business logic beyond removing deprecated enum values from types/constants
- No unrelated schema changes
- No refactors outside enum recreation + dependent type updates

### No New Concepts Rule

Do not add new enum values. Only remove the specified deprecated values.

### Production Ready Definition

- TypeScript no longer references removed values
- Migration applies in staging cleanly
- Verification queries prove no data loss and removed values cannot be inserted
- `pnpm lint`, `pnpm typecheck`, `pnpm build` pass; `pnpm test` passes except known baseline guardrail (if still present)

## Pre-Migration Checklist (Must Pass)

- Ticket 21.2 is complete: no runtime writes of `one_to_one`
- No runtime writes of the deprecated `learning_signal_type` values
- `pnpm typecheck` passes after removing deprecated values from TS unions/constants **before** running SQL
- SQL pre-flight queries return zero rows containing deprecated values (must be run and included in PR notes)

## Migration Requirements (Exact)

- Must run in a **single transaction**
- Use the pattern:
  - create enum v2 → alter columns via `USING` cast → drop old enum → rename v2
- Rebuild dependent indexes if required
- Include rollback steps in migration comments
- Add rollback runbook:
  - `docs/runbooks/enum-recreation-rollback.md`

### Required Pre-Flight SQL (Include In PR Notes)

- Count deprecated values in relevant tables (must be zero)
- Count `one_to_one` in match tables (must be zero)

### Required Post-Flight SQL (Include In PR Notes)

- Verify row counts unchanged
- Verify inserting removed values fails with enum error

## Deliverables

- `supabase/migrations/YYYYMMDD_recreate_enums_remove_deprecated.sql`
- `docs/runbooks/enum-recreation-rollback.md`
- Updated TS types/constants removing deprecated values
- PR description includes pre/post verification query results (staging)

## Verification Checklist

### Local

- `supabase db reset --no-seed`
- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`
- `pnpm build`

### Staging (Mandatory)

- `supabase db push --linked`
- Run pre-flight queries (must be zero)
- Apply migration
- Run post-flight queries + insert-failure checks

## Completion Criteria

- Deprecated enum values cannot be inserted
- Migration applies cleanly in staging with no data loss
- TypeScript types/constants no longer reference removed values
- Rollback procedure is documented
