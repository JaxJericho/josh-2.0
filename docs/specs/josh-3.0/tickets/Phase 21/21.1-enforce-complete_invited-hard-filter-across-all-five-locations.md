# Ticket 21.1 — Enforce complete_invited Hard Filter Across All Five Locations (L)

## Role

Principal Systems Architect & Full-Stack Engineer (matching safety, query correctness, production hardening)

## Phase

Phase 21 — Matching System Updates, Destructive Migrations, And Legacy Removal

## Complexity

Large

## What This Ticket Builds

A **hard, query-level exclusion** that guarantees profiles with `state = complete_invited` are **never** eligible as match candidates (and cannot initiate LinkUps), enforced across **five audited locations**.

## Why It Matters

`complete_invited` must not leak into matching under any circumstances (even if `is_complete_mvp = true`). This is safety-critical.

## Scope Locks (Anti-Scope-Creep)

### Allowed Change Surface

Only modify these areas (and only as needed for this ticket’s intent):

- `scripts/run-matching-job.mjs`
- `packages/core/compatibility-score-writer.ts`
- Candidate-fetch SQL used by matching/orchestration (existing SQL/functions only; no new tables)
- Match candidate import queries (existing path only)
- Existing initiator eligibility check codepath (no new endpoints)

### Forbidden

- No new tables, no new endpoints, no new background jobs, no new dashboards/UI, no new message flows, no new configs/flags.
- No refactors unrelated to adding the hard filters + guard.
- No new “matching modes” or new eligibility rules beyond blocking `complete_invited`.

### No New Concepts Rule

Do not introduce new enums, states, modes, or abstractions. If something seems missing, treat it as a spec change request, not an implementation.

### “Production Ready” Definition

Production ready means:

- Filter is wired at the query boundary in all five locations
- Defense-in-depth guard exists
- Tests cover the required behavior
- `pnpm lint`, `pnpm typecheck`, `pnpm build` pass; `pnpm test` passes except the known baseline guardrail if it still exists

## Behavioral Constraints

- The filter must be **query-level**, not in-memory filtering.
- Add the exact comment in each location:
  - SQL: `-- complete_invited hard filter: never relax`
  - TS/JS: `// complete_invited hard filter: never relax`
- Add **defense-in-depth**: application-level secondary guard in the matching job (lines 90–96) that detects any `complete_invited` candidate and excludes it (log/throw per existing conventions).
- Do not change unrelated matching semantics.

## Scope (All Five Locations Must Be Updated)

The filter must appear in all five audited locations:

1. `scripts/run-matching-job.mjs`
   - Add `.neq("state", "complete_invited")` to candidate pool query at **line 34**
   - Add defense-in-depth check at **lines 90–96**
   - Add comment near both

2. `packages/core/compatibility-score-writer.ts`
   - Add explicit candidate exclusion to any candidate fetch query
   - Add comment

3. Orchestration SQL candidate fetch
   - Add `state != 'complete_invited'` to candidate selection WHERE clause
   - Add comment

4. `match_candidates` import queries
   - Add explicit exclusion at query level
   - Add comment

5. Initiator eligibility check
   - Enforce `complete_invited` cannot initiate LinkUps at the eligibility/query boundary

If any paths differ in the repo, locate the true call sites via grep and apply the filter at the real candidate-fetch boundaries.
