# Ticket 22.3 — Remove Waitlist And Region Gate Runtime Paths

## Phase And Context

Phase 22 cleanup. This ticket removes all runtime code that checks waitlist status, enqueues waitlist activations, or gates access based on region enrollment. The 3.0 BETA model has no waitlist. Region state is now only consulted inside `evaluateEligibility({ action_type: 'can_initiate_linkup' })`. Every other runtime path that touches waitlist or region gate logic is dead code and must be removed.

**This is a deletion-heavy ticket.** The correct sequence is: grep first, confirm, then delete. Do not delete anything that has not been confirmed as dead by grep output.

## What To Remove

### Conversation Handlers And Router

The routing spec (Conversation Routing And Intent Detection Contract) explicitly states:

> "Waitlist gating and region-based access control are deprecated. There is no waitlist check in the routing flow."

Look for any of the following patterns in handler and routing files and remove them:

- Checks against `users.waitlist_status` or similar fields
- Any code path that sends a "you're on the waitlist" message
- Any code path that checks whether a region is "accepting new users" before routing
- Any code that reads a `waitlist_position` value to determine routing

Do not remove region assignment code. Users still have a `region_id`. Only remove code that uses region state to gate access outside of `evaluateEligibility`.

### Runners And Cron Jobs

Look in `scripts/` for any runner that:

- Activates waitlisted users (moves them from waitlisted → active based on capacity)
- Sends "your region is now open" notifications
- Processes a waitlist queue

Remove the entire runner if its only purpose is waitlist processing. If it also handles other work, remove only the waitlist portions and leave the rest intact. Document what you removed and why.

### Region Gate Checks Outside evaluateEligibility

Any code that does something like:

```typescript
const region = await getRegion(user.region_id);
if (!region.is_open) {
  // deny access
}
```

...in a conversation handler, router, or utility **outside of `evaluateEligibility()`** must be removed. The single allowed location for this check is inside `evaluateEligibility({ action_type: 'can_initiate_linkup' })`.

---

## What Not To Remove

- The `regions` table and its TypeScript type — region assignment still exists
- `user.region_id` column and any code that assigns or reads it for non-gating purposes
- `evaluateEligibility()` itself — it is the 3.0 gating mechanism
- The `can_initiate_linkup` eligibility check inside `evaluateEligibility()`, which legitimately reads `region.status`
- Any waitlist-related database migration files — these are historical and must not be touched
