# Ticket 22.1 — handleNamedPlanRequest

## Phase And Context

Phase 22. This is the first of two new conversation handlers in this phase. It implements the NAMED_PLAN_REQUEST path — when a user texts something like "See if Jordan is free for a hike Saturday", JOSH resolves the named contact from the user's Plan Circle, creates a plan brief, and enters a confirmation state.

**Dependencies that must be complete before writing any code:**

- Phase 16: `contact_circle` and `plan_briefs` tables exist in the schema
- Phase 17: `evaluateEligibility()` exists and accepts `{ userId, action_type: 'can_initiate_named_plan' }`
- Phase 19: The intent router exists and routes `NAMED_PLAN_REQUEST` intents to a handler registry. The LLM classifier already extracts `contactNames`, `activityHint`, and `timeWindowHint` fields on this intent type.

## Logic — Step By Step

### Step 1: Eligibility Check

```typescript
const eligibility = await evaluateEligibility({
  userId,
  action_type: "can_initiate_named_plan",
});

if (!eligibility.eligible) {
  await sendSms(
    userId,
    buildSubscriptionPrompt(eligibility.reason),
    correlationId,
  );
  // Do NOT update session mode — leave it unchanged (idle or wherever it was)
  return;
}
```

The subscription prompt must follow JOSH voice rules from the Conversation Behavior Spec:

- Warm, honest, no guarantees
- No jargon (no "eligibility", "subscription tier", "plan feature")
- Example tone: "To coordinate plans with your contacts, you'll need an active JOSH subscription. Reply SUBSCRIBE to get started."

Use an existing subscription message constant if one exists in `packages/core/src/messages.ts` (or equivalent). If not, define one there rather than inlining it.

### Step 2: Contact Resolution

The LLM classifier provides `contactNames: string[]` from the intent. Take `contactNames[0]` as the primary contact to resolve (single-contact plans only at MVP).

```typescript
const contactName = intentFields.contactNames?.[0];

if (!contactName) {
  // Router should not have dispatched here without a contact name, but handle defensively
  await sendSms(userId, CLARIFY_CONTACT_MESSAGE, correlationId);
  return;
}

const contact = await db
  .from("contact_circle")
  .select("id, contact_name, contact_phone_e164")
  .eq("user_id", userId)
  .ilike("contact_name", contactName) // case-insensitive match
  .maybeSingle();
```

**If contact is not found:**

```typescript
if (!contact) {
  await sendSms(
    userId,
    buildContactNotFoundMessage(contactName),
    correlationId,
  );
  // Session stays idle — this is not a terminal error, it's a soft miss
  return;
}
```

The "contact not found" message must:

- Name the contact the user mentioned (use the raw name they typed, not a normalized version)
- Offer to add them: "I don't have [name] in your contacts yet. Want to add them? Reply with their number."
- Stay within 1–2 sentences per JOSH voice rules

Define `buildContactNotFoundMessage(contactName: string): string` as a pure function in the handler file or in `packages/core/src/messages.ts`. It must not be hardcoded inline.

### Step 3: Create Plan Brief

```typescript
const planBriefId = crypto.randomUUID();

const { error: briefError } = await db.from("plan_briefs").insert({
  id: planBriefId,
  creator_user_id: userId,
  activity_key: intentFields.activityHint ?? null,
  proposed_time_window: intentFields.timeWindowHint ?? null,
  notes: null,
  status: "draft",
  created_at: new Date().toISOString(),
  updated_at: new Date().toISOString(),
});

if (briefError) {
  throw new Error(`Failed to create plan_briefs row: ${briefError.message}`);
}
```

Errors here are not swallowed — they propagate up to the router's error boundary, which handles structured logging and fallback messaging.

### Step 4: Update Session Mode

```typescript
await updateConversationSession(userId, {
  mode: "pending_plan_confirmation",
  state_token: `plan_brief:${planBriefId}:contact:${contact.id}`,
  updated_at: new Date().toISOString(),
});
```

The `state_token` format encodes both the plan brief ID and the resolved contact ID so the subsequent `SOCIAL_CHOICE` handler (when the user replies to confirm) can load context without an additional lookup. Confirm the `updateConversationSession` utility exists in the codebase before calling it. If the pattern differs, match existing convention.

### Step 5: Send Confirmation Prompt

```typescript
await sendSms(
  userId,
  buildPlanConfirmationPrompt({
    contactName: contact.contact_name,
    activityHint: intentFields.activityHint,
    timeWindowHint: intentFields.timeWindowHint,
  }),
  correlationId,
);
```

`buildPlanConfirmationPrompt` is a pure function that constructs a JOSH-voice confirmation message. It must:

- Name the contact and activity naturally, not as fields
- Ask for one confirmation from the user before anything is sent to the contact
- Stay within 2–3 sentences
- Example: "Got it — I'll reach out to Marcus about a hike on Saturday. Should I go ahead?"

Define this function in `packages/core/src/messages.ts` alongside other message builders.
