# Ticket 22.5 — Drop profiles.fingerprint Column

## Phase And Context

Phase 22 cleanup. This is the second step of the two-step fingerprint column rename. Phase 16 added the `coordination_dimensions` shadow column alongside `fingerprint` and copied all data. Phase 18 switched all reads and writes to `coordination_dimensions` via the `PROFILE_DIMENSIONS_COLUMN` constant. This ticket drops the old column and removes the forwarding constant.

This ticket must not run until all three conditions are confirmed:

1. All application code reads from and writes to `coordination_dimensions`
2. All rows in `profiles` have non-null `coordination_dimensions` values
3. No active code references `fingerprint` by string or constant

---

## Pre-Work: Required Checks Before Writing The Migration

**Check 1 — No TypeScript code references "fingerprint" as a column name:**

```bash
grep -rn "fingerprint" --include="*.ts" packages/
```

Acceptable remaining occurrences: comments explaining the historical rename, the `PROFILE_DIMENSIONS_COLUMN` constant itself (to be removed by this ticket), documentation files.

Not acceptable: any db query that selects, inserts, or updates `fingerprint` as a column; any TypeScript type that declares `fingerprint` as a property.

**Check 2 — PROFILE_DIMENSIONS_COLUMN is set to 'coordination_dimensions':**

```bash
grep -rn "PROFILE_DIMENSIONS_COLUMN" --include="*.ts" packages/
```

Confirm the constant value is `'coordination_dimensions'` and that all call sites use the constant rather than the literal string `'fingerprint'`.

**Check 3 — All profiles rows have non-null coordination_dimensions:**

Run this query against both staging and production before running the migration:

```sql
SELECT COUNT(*) FROM profiles WHERE coordination_dimensions IS NULL;
```

Must return 0. If any rows are null, re-run the Phase 16 data backfill before proceeding. Do not run this migration until the count is 0 in both environments.

---

## Migration File

**File:** `supabase/migrations/YYYYMMDD_drop_fingerprint_column.sql`

Replace YYYYMMDD with the actual execution date.

```sql
-- Drop legacy fingerprint column from profiles
-- Pre-condition: all application code uses coordination_dimensions
-- Pre-condition: SELECT COUNT(*) FROM profiles WHERE coordination_dimensions IS NULL = 0
-- Completes the two-step column rename begun in Phase 16

ALTER TABLE profiles DROP COLUMN fingerprint;
```

There is no automated rollback for this migration. The data exists in `coordination_dimensions`. In an emergency, `fingerprint` can be re-added and populated from `coordination_dimensions` — but this is a last resort only and must be documented as such.

---

## Code Update: Remove PROFILE_DIMENSIONS_COLUMN Constant

**File:** `packages/core/src/profile/profile-writer.ts`

Find the `PROFILE_DIMENSIONS_COLUMN` constant (introduced in Phase 18 as a forwarding mechanism during the transition period):

```typescript
export const PROFILE_DIMENSIONS_COLUMN = "coordination_dimensions";
```

**Step 1:** Identify all call sites:

```bash
grep -rn "PROFILE_DIMENSIONS_COLUMN" --include="*.ts" packages/
```

**Step 2:** For each call site, replace the constant reference with the literal string:

```typescript
// Before
.select(PROFILE_DIMENSIONS_COLUMN)

// After
.select('coordination_dimensions')
```

**Step 3:** After all call sites are updated, delete the constant declaration from `profile-writer.ts`.

**Step 4:** Run typecheck before touching the database:

```bash
pnpm typecheck
```

Typecheck must pass before the migration is applied.
