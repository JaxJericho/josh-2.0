# Ticket 22.2 — handlePostActivityCheckin: Solo Follow-Through Loop

## Phase And Context

Phase 22. This handler closes the solo coordination loop: after JOSH suggests a solo activity and the event time passes, a scheduled message sets the session to `post_activity_checkin` mode. When the user replies, this handler processes their response across up to three conversational steps, writes the appropriate `learning_signals` rows, and returns the session to `idle`.

This handler is a **multi-step state machine** within a single conversation mode. Unlike most handlers that complete in one exchange, this one may take 2–3 replies to reach a terminal state. The `state_token` tracks position within the checkin flow.

**Dependencies that must be complete before writing any code:**

- Phase 16: `learning_signals` table exists; solo signal type enum values added (`solo_activity_attended`, `solo_activity_skipped`, `solo_do_again_yes`, `solo_do_again_no`, `solo_bridge_accepted`)
- Phase 16: `plan_briefs` table exists with `activity_key` column
- Phase 17: `evaluateEligibility()` exists (used for bridge offer only, not for the checkin itself)
- Phase 19: Router dispatches to this handler when `session.mode = 'post_activity_checkin'`

## State Machine Within The Handler

The checkin flow has three possible steps. `state_token` encodes the current step.

```
state_token: 'checkin:awaiting_attendance:{plan_brief_id}'   → Step 1
state_token: 'checkin:awaiting_do_again:{plan_brief_id}'     → Step 2
state_token: 'checkin:awaiting_bridge:{plan_brief_id}'       → Step 3 (conditional)
```

**Step 1 — Did you go?**
Entry point. Session arrives here from the scheduler that sends the checkin prompt after the planned event time passes.

- Positive response ("yeah", "I went", "it was great") → write `solo_activity_attended` → advance to Step 2
- Negative response ("no", "didn't make it", "skipped it") → write `solo_activity_skipped` → set session to `idle` (terminal)
- Ambiguous response → local parse fails → ask one clarifier: "Did you end up going?" → keep state_token unchanged

**Step 2 — Would you do it again?**
Only reached after `solo_activity_attended` is written.

- Positive response ("yes", "definitely", "for sure") → write `solo_do_again_yes` → advance to Step 3 (bridge offer)
- Negative response ("no", "not really", "was fine once") → write `solo_do_again_no` → set session to `idle` (terminal)
- Ambiguous → ask one clarifier: "Worth doing again?" → keep state_token unchanged

**Step 3 — Bridge offer**
Only reached after `solo_do_again_yes` is written.

- The bridge offer message asks: "Want me to find someone to do this with next time?"
- Positive response → write `solo_bridge_accepted` → set session to `idle` (terminal)
- Negative/indifferent response → no signal written → set session to `idle` (terminal)
- No clarifier at this step — the bridge offer is optional and low-stakes. Any non-positive response is treated as a no.

### Local Parsing

Do not call the LLM for this handler. All parsing is local. Use a shared utility if one exists; otherwise implement inline:

```typescript
function parseAttendanceResponse(
  message: string,
): "positive" | "negative" | "ambiguous" {
  const normalized = message.trim().toLowerCase();
  const positivePatterns =
    /^(yes|yeah|yep|yup|went|did|i did|attended|it was|totally|definitely|sure|great|good|loved|liked)/;
  const negativePatterns =
    /^(no|nope|didn't|did not|skipped|couldn't|couldn't make|nah|not really|missed)/;
  if (positivePatterns.test(normalized)) return "positive";
  if (negativePatterns.test(normalized)) return "negative";
  return "ambiguous";
}
```

Implement equivalent parsers for do-again (`parseDoAgainResponse`) and bridge (`parseBridgeResponse`). Keep them as pure functions at the bottom of the handler file or in a `parsing-utils.ts` file alongside other local parsers.

### Writing Learning Signals

Every signal write must include an idempotency key in the format specified by the Learning And Adaptation System spec:

```
idempotency_key: `ls:solo_checkin:{plan_brief_id}:{user_id}`
```

For the multi-step flow, each signal type gets a unique idempotency key:

```
ls:solo_checkin:attended:{plan_brief_id}:{user_id}
ls:solo_checkin:skipped:{plan_brief_id}:{user_id}
ls:solo_checkin:do_again_yes:{plan_brief_id}:{user_id}
ls:solo_checkin:do_again_no:{plan_brief_id}:{user_id}
ls:solo_checkin:bridge_accepted:{plan_brief_id}:{user_id}
```

Signal write function pattern:

```typescript
async function writeSoloSignal({
  userId,
  signalType,
  planBriefId,
  activityKey,
  correlationId,
}: WriteSoloSignalParams): Promise<void> {
  const idempotencyKey = `ls:solo_checkin:${signalType}:${planBriefId}:${userId}`;

  const { error } = await db.from("learning_signals").insert({
    id: crypto.randomUUID(),
    user_id: userId,
    signal_type: signalType,
    subject_id: planBriefId,
    value_bool: true,
    meta: { activity_key: activityKey },
    occurred_at: new Date().toISOString(),
    ingested_at: new Date().toISOString(),
    idempotency_key: idempotencyKey,
  });

  if (error?.code === "23505") {
    // Duplicate signal — idempotent, log and continue
    log.warn("handle_post_activity_checkin.duplicate_signal", {
      userId,
      correlationId,
      signalType,
      idempotencyKey,
    });
    return;
  }

  if (error) {
    throw new Error(`Failed to write learning signal: ${error.message}`);
  }
}
```

### Fetching Activity Key

The signal's `meta.activity_key` must be populated from the `plan_briefs` row. Fetch it once at the top of the handler (after extracting `planBriefId`), then pass it to every `writeSoloSignal` call:

```typescript
const { data: planBrief } = await db
  .from("plan_briefs")
  .select("activity_key")
  .eq("id", planBriefId)
  .maybeSingle();

const activityKey = planBrief?.activity_key ?? null;
```

If the plan brief is not found, do not abort — checkin can still proceed. Log a warning and pass `null` for `activityKey`. The signal row is still valid without it; the learning job handles sparse meta gracefully.

### Full Step Dispatch

```typescript
const step = extractStep(stateToken); // 'awaiting_attendance' | 'awaiting_do_again' | 'awaiting_bridge'

switch (step) {
  case "awaiting_attendance":
    await handleAttendanceStep(
      userId,
      message,
      planBriefId,
      activityKey,
      session,
      correlationId,
    );
    break;
  case "awaiting_do_again":
    await handleDoAgainStep(
      userId,
      message,
      planBriefId,
      activityKey,
      session,
      correlationId,
    );
    break;
  case "awaiting_bridge":
    await handleBridgeStep(
      userId,
      message,
      planBriefId,
      activityKey,
      session,
      correlationId,
    );
    break;
  default:
    log.error("handle_post_activity_checkin.unknown_step", {
      userId,
      correlationId,
      stateToken,
    });
    await setSessionIdle(userId, correlationId);
}
```

Each step function is a named private async function in the same file. They are not exported.
