## Context

After JOSH sends a solo activity suggestion via `handleOpenIntent`, the conversation session enters `awaiting_social_choice`. The user’s next reply is classified as `PLAN_SOCIAL_CHOICE` and routes here.

The user may:

- Accept the suggestion
- Decline it
- Modify it (“Actually, hiking instead”)
- Request an alternative (“Something else”)

This ticket implements the handler behavior, plan creation on acceptance, session transitions, and audit events.

## Goals

- Implement `handlePlanSocialChoice(userId, message, session)` and wire it to the `PLAN_SOCIAL_CHOICE` route.
- Handle accept/decline/modify/alternative requests deterministically.
- Create a `plan_briefs` record on acceptance.
- Enforce “3 alternative requests max” then gracefully end and return session to idle.
- Write audit-log domain events for transitions.

## Non-Goals

- Do not modify Supabase schema or add migrations.
- Do not change router ordering or intent classification (done in 19.1/19.2).
- Do not implement Plan Circle / Named Plan logic here.
- Do not generate new activity copy; suggestions must use catalog copy.

### Behavior

Parse the user’s response into one of:

- Accept
- Decline
- Modify
- Request alternative

#### Accept

- Create a `plan_briefs` row with `status = confirmed`.
- Transition session mode to `pending_plan_confirmation`.
- Send a confirmation message.

#### Decline

- Set session mode to `idle`.
- Send a natural acknowledgment:
  - Example copy: “No worries — reach out whenever you feel like doing something.”

#### Modify

- Update the pending suggestion based on user feedback.
- Resend an updated suggestion.
- Remain in `awaiting_social_choice`.

Constraints:

- Do not invent activity descriptions. Any suggestion text must use `activity_catalog.short_description` as the base copy.
- If “modify” implies a different activity, select a new activity from `suggestSoloActivity(userId)` using the modification as a constraint where possible (without schema changes).

#### Request Alternative

- Call `suggestSoloActivity(userId)` again, excluding the previous suggestion, resend, remain in `awaiting_social_choice`.
- Track a per-session “alternative request count”.
- After 3 alternative requests with no acceptance:
  - Set session mode to `idle`
  - Send a graceful message:
    - Example copy: “We can revisit this whenever you’re in the mood.”

Tracking note:

- Use existing session state storage (e.g., session `state_token` and/or existing session metadata/context) without schema changes.

### Audit Log Domain Events

All transitions must write domain events to the audit log using the repo’s existing audit/event logging mechanism.

Minimum events to record:

- Social choice accepted → plan confirmed
- Social choice declined → session idle
- Social choice modified → suggestion updated
- Alternative requested (count increment)
- Alternative limit reached → session idle

## Verification

- Reply “yes” to a suggestion:
  - `plan_briefs` row created with `status = confirmed`
  - session mode = `pending_plan_confirmation`
- Reply “no”:
  - session mode = `idle`
  - no `plan_briefs` row created
- Reply “actually hiking”:
  - suggestion updates
  - session remains `awaiting_social_choice`
- Three consecutive “something else” requests:
  - session returns to `idle` gracefully
